Googleログインとは

Firestoreとは

クラウド上にデータを置く箱

  • 記録データを置ける
  • ログインユーザーごとに分けられる
  • 今日のアプリは保存して読む

Firestoreは、Googleのクラウド上にデータを置くためのデータベースです。記録帳アプリなら、タイトル、本文、気分、作成日時などを置けます。

作業2: プロジェクトを作成

まだ、プロジェクトはないので作成します。

プロジェクトに名前を付ける

新しいプロジェクトを作る場合は、練習用だと分かる名前を付けます。たとえば、My DiaryやMy Noteのように、あとで見ても何のアプリ用か分かる名前がよいです。プロジェクト名に日本語が使えない場合があるため、英数字で分かりやすく付けると安全です。ここでは「My Diary」と入力して、「プロジェクト作成」をクリックします。

プロジェクト設定

Google Cloud projectに今作成したプロジェクトが設定されました。データが保存されるサーバはasia-east1を選びましょう。選べない場合は講師にお尋ねください。App support emailは自動で入力されており、編集不可能です。Database IDを自分用に控えることは後々大切ですが、画面共有や教材では公開しないでください。

Firebaseプロジェクトの接続先を選ぶ

  • Firebase / Auth 機能は Firebaseプロジェクト に紐づける
  • Firebase は Google Cloudプロジェクト を土台として動く
  • 既存プロジェクトがない場合は
    No Cloud Projects Available と表示される
  • 新しいアプリでは、基本的に 「プロジェクトを作成」 を選ぶ

この画面では、アプリに Firebase と Authentication、つまりログイン機能 を追加するために、接続先となるプロジェクトを選びます。

ここで出ている “Google Cloud project” という言葉が少し分かりにくいのですが、Firebase は Google Cloud のプロジェクトを土台として動きます。つまり、Firebase Authentication や Firestore などの機能は、アプリ単体に直接付くのではなく、Firebase プロジェクトに対して設定される と考えてください。

画面上部に “No Cloud Projects Available” と表示されています。これは、現在のGoogleアカウントで選択できる既存のCloudプロジェクトが見つかっていない、またはまだ明示的に選ばれていない状態を示しています。エラーというより、これから接続先を決める段階です。

選択肢としては、主に3つあります。 既に使いたいプロジェクトがある場合は 「プロジェクトをインポート」 を選びます。 新しくこのアプリ用にプロジェクトを作る場合は 「プロジェクトを作成」 を選びます。 また、環境によっては “Default project” のような既定のプロジェクト候補が表示されることもあります。

講義や演習では、設定が他のアプリと混ざらないように、基本的には アプリごとに新しいプロジェクトを作成する 方が分かりやすいです。プロジェクト名も、後でFirebase Consoleで見分けられるように、アプリ名や講義回が分かる名前にしておくとよいです。

この操作で決めているのは、あくまで Firebase機能をどのプロジェクトに紐づけるか です。Googleログインなど、具体的な認証方法の有効化は、この後にFirebase側で設定していきます。

メニューの セキュリティ にカーソルを合わせる、または展開すると、関連する機能が表示されます。ここには App Check、Authentication、Phone Verification などがあります。

注意点として、Phone Verification は電話番号確認に関する機能であり、今回設定したいGoogleログインとは別の項目です。また、App Check はアプリやAPIの不正利用対策に関する機能です。ログイン方法を設定する場合は、まず Authentication を開きます。

画面右上にはプロジェクト名 My Diary が表示されています。Firebaseの設定を行うときは、必ず自分が意図したプロジェクトを開いているか確認してください。別のプロジェクトで設定してしまうと、アプリ側からログイン機能が正しく使えない原因になります。

Googleログインを有効にする

  • Authentication の ログイン方法 タブ
    • ログイン方法は プロバイダ として管理される
    • Google のステータスが 有効 になっている
    • 未設定の場合は 「新しいプロバイダを追加」 からGoogleを追加
    • Firebase側の設定後、アプリ側にもログイン処理を追加する

この画面は、Firebase Authentication の ログイン方法 タブです。

ここでは、アプリで利用できるログイン方法、つまり ログインプロバイダ を確認・追加できます。ログインプロバイダとは、ユーザーがどの方法でログインできるかを表すものです。

画面では、Googleのプロバイダだけを確認します。講義では初心者の混乱を防ぐため、Googleログインだけを有効にします。

今回の目的はGoogleログイン認証ですので、特に確認するべきなのは Google の行です。Google のステータスが 有効 になっていれば、Firebase側ではGoogleログインを使う準備ができている状態です。

まだGoogleが表示されていない場合や、無効になっている場合は、右側の 「新しいプロバイダを追加」 からGoogleを選び、有効化します。

ただし、この画面でGoogleログインを有効にしただけでは、アプリ画面に自動でログインボタンが追加されるわけではありません。Firebase側でGoogleログインを許可したうえで、アプリ側にも「Googleでログインする」ボタンや、ログイン処理を追加する必要があります。

この画面は、Firebase側の認証設定が正しくできているかを確認する場所だと考えてください。

  • セキュリティルールとエラーハンドリングも反映

この画面では、Gemini がアプリに Firebase Firestore と Firebase Authentication を統合したことを報告しています。

Firestore は、アプリの記録データをクラウドに保存するためのデータベースです。これまではブラウザ上だけで扱っていた記録データを、Firebase のクラウドデータベースに保存できるようになった、ということです。

また、Googleアカウントによるログイン認証も追加されています。これにより、誰が記録を書いたのかを Firebase 側で識別できるようになります。ログインしたユーザーごとに、自分専用の記録を保存・表示する構成になります。

ここで重要なのは、AI が「実装が完了しました」と表示していても、まだ人間側での確認が必要だという点です。実際にログインできるか、記録が保存されるか、ログアウトできるか、別のユーザーの記録が見えないかを、順番に確認していきます。

この画面は、アプリを開いた直後のログイン画面です。

まだユーザーがログインしていないため、記録の入力画面ではなく、Googleアカウントでログイン というボタンが表示されています。

Firebase Authentication を使うと、このようにアプリにログイン機能を追加できます。今回のアプリでは、Googleアカウントを使ってログインします。ユーザーは新しいIDやパスワードを作る必要がなく、普段使っているGoogleアカウントでログインできます。

この段階では、まだ記録データの入力や一覧表示は行いません。まずログインして、Firebase が「この人は誰か」を確認します。その後に、そのユーザー専用の記録画面が表示されます。

画面下には、個人データがFirebaseの暗号化やセキュリティルールによって保護されることが示されています。ログイン機能を付ける目的は、単に画面をかっこよくすることではなく、ユーザーごとにデータを安全に分けて扱うためです。

表示されているアカウントとは別のアカウントを使いたい場合は、別のアカウントを使用 を選びます。講義や演習では、個人用アカウント、大学用アカウント、検証用アカウントなどを混同しないように注意が必要です。

ここで重要なのは、パスワードをアプリに直接入力しているわけではないという点です。本人確認はGoogle側で行われ、アプリはFirebase Authenticationを通じて、ログイン結果を受け取ります。

ログイン後の記録画面を確認する

この画面は、Googleログインが完了した後のアプリ画面です。

画面上部には、ログイン中のユーザー情報が表示されています。また、右側には サインアウト ボタンがあります。これが表示されていれば、Googleログインが正常に完了し、アプリ側がログイン状態を認識できていることが分かります。

アプリ本体には、記録のタイトル、本文、気分を入力する欄があります。入力した内容は、記録する ボタンを押すことで保存されます。

このアプリでは、保存先として Firebase Firestore を使っています。つまり、記録はブラウザの一時的なデータではなく、クラウド上のデータベースに保存されます。画面上部にクラウド保存の状態が表示されている点も確認できます。

また、Googleログインを使っているため、保存された記録はログインユーザーごとに分けて管理できます。自分のアカウントでログインした場合は自分の記録だけが表示され、別のユーザーの記録が見えないように、Firebaseのセキュリティルールで制御します。

Googleログインが単なる入口ではなく、誰のデータかを判定し、安全にクラウド保存するための仕組みであることを理解してください。

記録しようとするとエラーが出た

  • 「記録する」を押すとエラーが表示された
  • 原因は Firestore Security Rules の可能性が高い
  • ログインできることと、書き込めることは別
  • エラー画面を見ながら、AIに状況を具体的に伝える
  • ルールを広げすぎず、最小限の修正にする

この画面では、「記録する」ボタンを押したあとに、記録の作成に失敗したことが表示されています。

エラーメッセージには、セキュリティ規則を確認してください と書かれています。ここでいうセキュリティ規則とは、Firestore の Security Rules のことです。

大事なポイントは、Googleログインに成功していること と、Firestoreに書き込めること は別だという点です。ログインできていても、Firestore側のルールで書き込みが許可されていなければ、記録は作成できません。

このようなときは、やみくもにルールを変更するのではなく、まずAIに状況を具体的に伝えます。

指示の後にエラーメッセージを直接付記しているところがポイントです。与える情報が正確な方が、良好な結果が得られます。

「記録の作成に失敗しました。セキュリティ規則を確認してください」
というエラーが出ます。

Firestoreの保存先は users/{ログインしたユーザーID}/entries です。

Cloud Billing、Blaze、Cloud Run、Firebase App Hosting、

ログインした本人だけが、自分の記録を作成できるように、

公開書き込みや allow read, write: if true; のような広すぎるルールにはしないでください。

  • アプリのビルドも正常に完了
  • 認証条件とFirestoreルールのずれを調整
  • 自分の記録だけ操作できる条件を維持

画面の Action history には、firestore.rules を編集し、ビルドが完了したことが表示されています。つまり、アプリ側だけでなく、Firestore のアクセス制御にも変更が入ったということです。

ここで重要なのは、AIが修正してくれたからといって、そのまま無条件に信用するのではなく、何が変わったのかを確認する ことです。Security Rules は、データベースの入口を守るルールです。広げすぎると、他のユーザーからデータを読まれたり、書き換えられたりする危険があります。

この画面では、「自分自身の記録を参照・追加・削除・更新できる」と説明されています。教材では、ここを次のように言い換えると分かりやすいです。

「ログインしているユーザー本人だけが、自分の場所にある記録を扱えるようにした、という意味です。」

ただし、削除や更新まで許可するかどうかは、授業回の到達点に合わせて判断します。記録の作成だけを扱う回であれば、削除や更新まで無理に入れる必要はありません。

講義では、ここで次の注意をはっきり伝えます。

Security Rules を修正するときは、公開書き込みにしない。 テストモードに戻さない。 課金、Deploy、Publish、Cloud Run、App Hosting には進まない。

修正後、記録が表示された

  • 記録の一覧に新しいカードが表示された
  • タイトル、本文、気分が反映されている
  • Firestoreへの書き込みと読み取りが通った
  • 再読み込み後も残るか確認する
  • 別ユーザーの記録が見えないことも確認する
先ほどは「記録の作成に失敗しました」というエラーが出ていましたが、Firestore のルールとアプリ側の処理を調整したことで、記録が作成できるようになりました。

一覧の上には、記録数が 1 と表示されています。カードの中には、気分、タイトル、本文が表示されています。これにより、入力した内容がアプリ内で記録として扱われていることを確認できます。

ただし、この画面だけで確認を終わらせないようにします。次の確認も行います。

まず、ブラウザを再読み込みしても記録が残っているかを確認します。再読み込み後も表示されれば、画面内だけの一時的な表示ではなく、Firestoreから読み込めている可能性が高くなります。

次に、サインアウトして再ログインしたときにも、自分の記録が表示されるかを確認します。さらに、別のアカウントでログインした場合に、この記録が見えないことも重要です。

ここで伝えたいのは、アプリが動いたかどうかだけではありません。Firestoreを使う場合は、保存できること と 他人のデータが見えないこと の両方を確認する必要があります。

表示されないときに見る場所

  • ログインしているか
  • UIDとデータの置き場所が合っているか
  • Security Rulesが合っているか

保存した記録が表示されない場合は、順番に確認します。まずログインできているか。次に、Firestoreに置いた場所がログインユーザーのUIDと合っているか。最後にSecurity Rulesが正しいかを確認します。
