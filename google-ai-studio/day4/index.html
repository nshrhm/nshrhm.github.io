Security Rulesの変更を確認する

今日は全部は読みません

見るところは3つだけです。

  • 最初は 全部禁止
  • records だけにルールを付ける
  • ログイン本人の記録だけ読む・作る

この画面は、Cloud FirestoreのSecurity Rulesが変更されたところです。

左側には、以前のルールが表示されています。 右側には、現在の新しいルールが表示されています。

ここで大事なのは、コードを1行ずつ全部読むことではありません。 このルールは、初心者向けの授業で逐語的に説明するには長すぎます。

今日見るポイントは3つだけです。

1つ目は、最初のルールは「全部禁止」だったということです。 最初の状態では、誰も読めない、誰も書けない、という安全側の設定になっています。

2つ目は、現在のルールでは records というコレクションだけに、具体的な許可条件が付いているということです。 Firestoreのすべてを自由に読めるようにしているわけではありません。

3つ目は、ログインしている本人の記録だけを読める、作れるようにしているということです。 そのために、ログイン状態と、記録の中にある userId を見ています。

この画面では、英語のコメントや関数がたくさんあります。 しかし、今日の授業では、細かい文法を覚える必要はありません。 Security Rulesは、Firestoreの入口で「誰が読めるか、誰が書けるか」を決めるものだと理解できれば十分です。

また、この画面にプロジェクトID、Database ID、メールアドレスなどが映る場合は、教材化する前に必ずぼかしてください。 Security Rulesの内容自体は説明に使ってよいですが、個人やプロジェクトを特定できる情報は見せません。

最初のルールは「全部禁止」

allow read, write: if false;

  • これは、 誰も読めない・誰も書けない という意味です。
  • 安全だが、アプリから保存できません。

まず、初期ルールを見ます。

初期ルールの中には、allow read, write: if false; という行があります。

これは、とても簡単に言うと、「読むことも、書くことも許可しない」という意味です。

read は読むことです。 write は書くことです。 if false は、条件が常に成り立たない、という意味です。

つまり、誰がアクセスしても、読み取りも書き込みもできません。

これは、安全側の設定です。 しかし、このままだと、アプリからFirestoreに記録を保存できません。 第3回で「記録の作成に失敗しました。セキュリティ規則を確認してください」というエラーが出た理由は、このようにFirestore側が書き込みを止めていた可能性があります。

ここで大切なのは、エラーが出たからといって、すぐに全員に開放するルールにしないことです。 安全に止まっている状態から、必要な範囲だけを少し開ける、という考え方で進めます。

現在のルールで見るところ

長いルールの中で、今日見るのはここだけです。


　match /records/{recordId} {
　  allow read: if isSignedIn()
　              && resource.data.userId == request.auth.uid;

　  allow create: if isEmailVerified()
　                && isValidRecord(request.resource.data);
　}

records の記録だけを対象にする

現在のルールは長く見えますが、今日見るところは records の部分だけです。

match /records/{recordId} というところは、Firestoreの records コレクションに入っている記録を対象にする、という意味です。

今回のアプリでは、Firestoreの中に records というコレクションがあります。 その中に、記録1件ごとのドキュメントが入っています。

この records に対して、読むときの条件、作るときの条件、更新するときの条件、削除するときの条件が書かれています。

第4回で中心に見るのは、read と create です。 read は、Firestoreから記録を読むことです。 create は、新しい記録を作ることです。

このスライドでは、update や delete の細かい条件は扱いません。 実際のルールには update や delete の行もありますが、第4回では編集機能や削除機能の新規追加はしません。 削除は第5回以降で、必要な場合だけ扱います。

今日の理解としては、records という記録の箱に対して、「誰が読めるか」「誰が新しく作れるか」を決めている、と考えてください。

ログインしている人だけ

isSignedIn()

中では、次のような確認をしています。

request.auth != null

これは、 Googleログイン済みならOK という意味です。

次に、ログインしている人だけ、という条件を見ます。

現在のルールでは、isSignedIn という関数が使われています。 関数という言葉は難しく聞こえるかもしれませんが、ここでは「名前を付けた確認項目」くらいに考えてください。

isSignedIn の中では、request.auth != null という確認をしています。

request.auth は、今アクセスしている人のログイン情報です。 これが null、つまり空っぽでないなら、誰かがログインしているという意味になります。

初心者向けには、request.auth != null は「Googleログイン済みならOK」と説明します。

ここで注意するのは、ログインしているだけでは、どの記録でも読めるわけではないということです。 ログイン済みかどうかは、最初の確認です。

このあと、もう1つ大事な確認があります。 それが、「その記録が本当に本人のものか」という確認です。

自分の記録だけ読む

resource.data.userId == request.auth.uid

保存済みの userId = 今ログインしている人のID

のときだけ読む、という意味です。

次に、自分の記録だけ読むための条件を見ます。

resource.data.userId == request.auth.uid という行があります。

少し長いですが、意味はシンプルです。

resource.data は、すでにFirestoreに保存されている記録の中身です。 その中の userId は、この記録が誰のものかを表す印です。

request.auth.uid は、今ログインしている人のIDです。

つまり、この行は、保存済みの記録の userId と、今ログインしている人のIDが一致するかを確認しています。

一致すれば、自分の記録として読めます。 一致しなければ、他人の記録なので読めません。

この仕組みによって、records という同じコレクションの中に記録が入っていても、自分の記録だけを表示できます。

自分の記録だけ作る

保存時も、userId を確認します。

data.userId == request.auth.uid

他人の userId では保存できない ということです。

読み取りだけでなく、作成するときにも userId を確認しています。

ルールの中には isValidRecord という関数があります。 この関数は、これから保存しようとしている記録の形が正しいかを確認しています。

title があるか。 content があるか。 mood が決められた選択肢になっているか。 createdAt が正しい時刻になっているか。 そして、userId が今ログインしている人のIDと一致しているか。

今日、細かい条件を全部覚える必要はありません。 大事なのは、保存するときにも「これは本人の記録か」を確認していることです。

たとえば、誰かが他人の userId を入れて記録を作ろうとしても、ルールで止める考え方です。

アプリ画面で自分の記録だけを表示することも大切ですが、それだけでは不十分です。 Firestore側のSecurity Rulesでも、本人の記録だけを作れるようにしておく必要があります。

広すぎるルールにしない

これは使いません。

allow read, write: if true;

これは、 誰でも読める・誰でも書ける に近い危険な設定です。

最後に、広すぎるルールにしないことを確認します。

Firestoreのエラーが出たときに、AIが allow read, write: if true; のようなルールを提案することがあります。

これは、簡単に言うと、誰でも読める、誰でも書ける、に近い危険な設定です。

たしかに、このようなルールにすると、エラーは消えるかもしれません。 しかし、それは安全になったという意味ではありません。 むしろ、Firestoreの入口を大きく開けすぎてしまいます。

この講座では、公開読み書きのような広すぎるルールは使いません。

今日の考え方は、次の3つです。

まず、何も指定していない場所は閉じる。 次に、records だけにルールを付ける。 最後に、ログイン本人の記録だけを読める・作れるようにする。

別アカウント確認

講師の指示がある場合のみ確認します。

  • 別アカウントでは自分の記録が見えない
  • 空状態なら自然
  • メールアドレスやUIDは写さない

権限確認として、講師の指示がある場合のみ、別アカウントでの見え方を確認します。別のアカウントでログインしたときに、先ほど作った自分の記録が見えなければ、ユーザーごとに記録が分かれている可能性が高いです。まだ記録がないアカウントでは、空状態メッセージが出るのが自然です。この確認では、メールアドレス、UID、プロフィール画像をスクリーンショットに写さないようにしてください。受講生が無理に複数アカウントを作る必要はありません。

保存できないときの聞き方


Googleログイン後に「記録する」を押してもFirestoreに保存できません。
エラー内容は次のとおりです。

（ここにエラー文を貼ります）

ログインした本人だけが自分の記録を作成できる最小限の修正にしてください。

Cloud Billing、Blaze、Cloud Run、Firebase App Hosting、Deploy、Get API key は使わないでください。
公開書き込みや allow read, write: if true; のような広すぎるルールにはしないでください。

保存できないときは、エラー文をAIに正確に伝えることが大切です。エラー文を貼り付けたうえで、「最小限の修正」で済ませるように明記します。そして、Cloud Billing、Blaze、Cloud Run、Firebase App Hosting、Deploy、Get API keyは使わないこと、公開書き込みやallow read, write: if true;のような広すぎるルールにしないことを明記します。AIへの質問は、原因を当てさせるより、条件を守って最小限で直してもらう形にします。

再読み込み後に出ないとき


Firestoreに保存できたように見えますが、ブラウザを再読み込みすると記録が表示されません。
ログイン状態、Firestoreの保存先、読み込み処理、Security Rulesの順に、確認すべき点を教えてください。

Cloud BillingやBlazeへのアップグレードは行いません。

見る順番を指定すると、確認しやすくなります。

再読み込み後に記録が表示されない場合は、いきなりルールだけを変えないようにします。まずログイン状態を確認します。次に、Firestoreの保存先がアプリの読み込み先と一致しているかを見ます。その次に、読み込み処理が動いているかを確認します。最後にSecurity Rulesを確認します。この順番をAIに指定すると、確認作業が整理されます。Cloud BillingやBlazeへのアップグレードは行わないことも、必ず書いておきます。

ConsoleやRulesで迷ったとき


Firebase Consoleで、保存した記録がどこにあるか分かりません。
Firestore Consoleで確認する手順を初心者向けに説明してください。



Firestore Security Rulesが広すぎないか確認したいです。

という条件になっているか確認してください。

Firebase Consoleでデータが見つからないときは、保存場所の構造を前提にしてAIへ聞きます。users、ログインしたユーザーID、entries、記録IDという階層を指定すると、確認手順を説明してもらいやすくなります。また、Security Rulesが広すぎないか不安なときは、ログインした本人だけ、自分の記録だけ、読み取りと作成だけ、という条件になっているか確認してもらいます。allow read, write: if true;のような公開設定にはしない、と明確に伝えます。

第4回 終了チェック

最後に、第4回の終了チェックです。第3回で作った「マイ記録帳」を開けたか。講座用のプライベートGmailでログインしているか。架空のサンプル記録を1件作ったか。保存後に一覧に表示され、再読み込み後も表示されたか。Firebase Consoleで対象プロジェクトとFirestoreの構造を確認したか。Security Rulesが誰が読めるか・書けるかを決めるものだと説明できるか。request.auth != nullとrequest.auth.uid == userIdの意味を説明できるか。公開読み書きにしていないか。課金、公開、APIキー作成に進んでいないかを確認します。
