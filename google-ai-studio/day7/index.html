エラーは失敗ではなく材料

エラーが出たら、次を集めます。

  • 何をしたか
  • どんな表示になったか
  • エラー文
  • 期待した動き
  • 使わない機能・進まない導線

材料が多いほど、AIに相談しやすくなります

今日の中心となる考え方です。エラーは失敗ではなく、AIに相談するための材料です。

たとえば、「記録する」ボタンを押した、保存失敗と表示された、エラー文には権限やSecurity Rulesのことが書かれていた、期待した動きはFirestoreに保存されて一覧に出ることだった、というように整理します。

さらに、Cloud Billing、Blaze、Cloud Run、Firebase App Hosting、Deploy、Publish、Get API keyを使わないこと、allow read, write: if true; のような広すぎるルールにしないことも一緒に伝えます。

AIには、現在の状態、期待する状態、守る条件をセットで渡します。これが、AIデバッグの基本です。

デバッグは「観察」から始める

原因を当てに行かない。

まず集めるもの

  • 直前にした操作
  • 画面に出た表示
  • エラー文
  • 期待した動き
  • 守る条件

第7回では、デバッグの考え方を新しく学びます。エラーが出たとき、すぐに原因を当てようとしなくて大丈夫です。まず観察します。どの操作をしたか、画面に何が出たか、エラー文は何か、期待していた動きは何か、そして守りたい安全条件は何かを集めます。AIに相談するときも、この材料があるほど、修正の方向が具体的になります。

AI Studioで見る4つの場所

  • Preview: 画面で何が起きたか
  • Chat: AIに状況を説明する
  • Code: 何が変わったかを見る
  • Firebase Console: ログイン・保存・ルールを見る

今日は、無理にコード編集しません。

Google AI Studio Buildでは、ただ画面を見て終わりではありません。Previewでは、実際にアプリを触って何が起きたかを確認します。Chatでは、AIに状況を説明して修正を依頼します。Codeでは、AIがどこかのファイルを変えていることを確認できます。ただし、今日は受講生がコードを無理に編集する必要はありません。Firebase Consoleでは、Googleログイン、Firestore、Security Rulesを確認します。この4つを役割分担して見ることが、今日の新しい知識です。

Checkpointは「戻り地点」

AI Studio Buildでは、前の状態へ戻れる場合があります。

  • AIに指示する
  • アプリが変更される
  • 変更後の状態が残る
  • 必要なら前のCheckpointへ戻す

大きく壊れたときの安全策です

ここで、Google AI Studio BuildのCheckpointを紹介します。Checkpointは、AIに指示してアプリを作っていく途中の戻り地点です。ゲームのセーブポイントに近いものだと考えてください。AIにプロンプトを送り、処理が終わると、その変更後の状態が履歴として残る場合があります。大きく壊れたときは、さらに修正を重ねる前に、前の動いていたCheckpointへ戻すことを考えます。ただし、何でも完全に元通りにする魔法ではありません。Firestoreの中の保存データやFirebase Consoleの設定まで全部戻るとは考えず、まずはアプリ編集の戻り地点として扱います。

小さく頼むほど、戻しやすい

1回のプロンプトを小さくします。

悪い例：

検索、編集、削除、デザイン、Rulesを全部直して

良い例：

保存後に入力欄を空にしてください

1回の変更を小さくすることがデバッグのコツです。

Checkpointを使いやすくするには、プロンプトを小さく分けることが大切です。1回のプロンプトで検索、編集、削除、デザイン、Security Rulesまで全部頼むと、どこで壊れたか分かりにくくなります。戻すときも、大きな変更をまとめて戻すことになります。反対に、1回のプロンプトを1つの変更に絞ると、もし壊れても、どの変更が原因だったか分かりやすくなります。今日は、AIに大きく任せるのではなく、小さく頼んで、見て、必要なら戻すという流れを学びます。

戻すか、直すか

状況 まず考えること
文字や余白だけ変えたい AIに最小修正を頼む
追加項目だけ一覧に出ない 診断して最小修正
ログイン・保存・読み込みがまとめて壊れた Checkpointへ戻す候補
AIが余計な機能を大量追加した Checkpointへ戻す候補
Security Rulesが広くなりすぎた Checkpointへ戻す候補

直し続けるより、戻した方が早いことがあります。

不具合が出たとき、いつもAIに直し続けてもらえばよいとは限りません。文字や余白のような小さなずれなら、AIに最小修正を頼めばよいです。追加項目が一覧に出ない程度なら、まず診断して、保存処理か表示処理のどちらかを見ます。一方で、ログイン、保存、読み込みがまとめて壊れた場合や、AIが余計な機能を大量に追加した場合、Security Rulesが広くなりすぎた場合は、前のCheckpointへ戻す方が安全なことがあります。第7回では、直すだけでなく、戻す判断も学びます。

戻す前にメモする

Checkpointへ戻す前に、短く記録します。


　直前に頼んだこと:
　壊れたところ:
　前に動いていたこと:
　戻したい理由:

メールアドレス、UID、プロジェクトIDは書きません。

Checkpointへ戻す前に、何が起きたかを短くメモします。直前にAIへ頼んだこと、壊れたところ、前に動いていたこと、戻したい理由を書きます。このメモは、AIに渡すためだけでなく、自分が状況を整理するためにも役立ちます。ただし、メールアドレス、UID、プロジェクトID、Database ID、APIキーのように見える文字列、実在の個人情報や業務情報は書きません。戻す前にメモしておくと、同じ失敗を繰り返しにくくなります。

Restore後も確認する

戻したら、もう一度確認します。

  • ログインできるか
  • 保存できるか
  • 読み込めるか
  • 追加項目が残るか
  • 危険導線に進んでいないか

戻しただけで安心しません。

Restoreを使って前のCheckpointへ戻したあとも、必ず自分で確認します。ログインできるか、保存できるか、読み込めるか、追加項目が残るかを見ます。また、Deploy、Publish、Get API key、Upgrade、Cloud Billing、Blaze、Cloud Run、Firebase App Hostingに進んでいないことも確認します。戻したつもりでも、画面やチャットの状態が自分の期待とずれていることがあります。デバッグでは、戻すことも確認の一部です。

Restore後の再出発プロンプト


　Checkpointで、前の動いていた状態に戻しました。
　直前の失敗した変更は採用しません。
　
　まず、現在の状態を前提に、まだコードやSecurity Rulesを変更しないでください。
　次に確認すべき点だけを、初心者向けに順番に教えてください。
　
　守る条件:
　- Googleログイン、Firestore保存、Firestore読み込みを維持する
　- ログインした本人の記録だけを扱う
　- allow read, write: if true; のような広すぎるルールにしない
　- Cloud Billing、Blaze、Cloud Run、Firebase App Hosting、Deploy、Publish、
　　Get API key は使わない

Checkpointで戻したあと、AIが前の失敗した指示に引きずられることがあります。そのため、再開するときは、直前の失敗した変更は採用しません、と明確に伝えます。そして、まだコードやSecurity Rulesを変更しないでください、と書きます。まず確認すべき点だけを順番に教えてもらいます。これにより、戻した直後にまた同じ失敗を繰り返すリスクを下げます。受講生には、このプロンプトをコピーして、必要なときに使えるようにします。

Checkpointの限界

次のようには考えません。

  • Firestoreデータの完全バックアップ
  • Firebase Console設定の完全バックアップ
  • 課金・公開状態の取り消しボタン
  • AIが失敗指示を完全に忘れる保証

大切な説明文やエラー情報は、別にメモします。

Checkpointは便利ですが、限界があります。教材では、Firestoreデータの完全バックアップ、Firebase Console設定の完全バックアップ、課金や公開状態の取り消しボタンとしては扱いません。また、Restoreしたあとに、AIが失敗した指示を完全に忘れる保証もありません。だからこそ、重要な状態、発表用の説明文、エラー文、AIに聞いた内容は、別にメモしておきます。初心者には、Checkpointは安全策の一つであり、確認とメモを置き換えるものではない、と伝えます。

AIに渡す情報の基本

「動きません」だけでは足りない。書くことは、

  1. 直前にした操作
  2. 画面に出た表示
  3. エラー文
  4. 期待する動き
  5. 守りたい安全条件

AIに相談するとき、「動きません」だけでは情報が足りません。

まず、直前にした操作を書きます。たとえば、Googleログインボタンを押した、記録するボタンを押した、ブラウザを再読み込みした、などです。

次に、画面に出た表示を書きます。ログイン画面のまま、保存失敗と出た、一覧が空のまま、追加項目だけ消えた、などです。

エラー文があれば、個人情報や識別子を確認したうえで貼ります。そして、期待する動きも書きます。最後に、Cloud Billingに進まない、広すぎるSecurity Rulesにしない、本人の記録だけを扱う、といった安全条件を入れます。

これにより、AIの修正範囲を小さくできます。

デバッグカードを書く


　1. 直前にした操作:
　2. 画面に出た表示:
　3. エラー文:
　4. 期待した動き:
　5. 守りたい条件:

「動きません」ではなく、材料を渡します。

今日の中心になる道具が、デバッグカードです。直前にした操作、画面に出た表示、エラー文、期待した動き、守りたい条件の5つを短く書きます。エラー文を貼る場合は、メールアドレス、UID、プロジェクトID、Database ID、APIキーのように見える文字列が入っていないか確認します。これをAIに渡すと、「動きません」だけよりも、はるかに具体的に相談できます。

症状を分類する

症状 まず見る場所
ボタンが反応しない 画面・クリック処理
Googleログインできない Authentication
保存失敗 保存処理・Security Rules
再読み込み後に消える 保存・読み込み
追加項目だけ消える 保存項目・一覧表示
Consoleが開かない アカウント・ブラウザ

原因を一つに決めつけません。

エラーは、まず症状で分類します。ボタンが反応しないなら、画面やクリック処理を見ます。GoogleログインできないならAuthenticationを見ます。保存失敗なら、保存処理とSecurity Rulesを見ます。再読み込み後に消えるなら、Firestoreに実際に保存されているか、読み込み処理があるかを見ます。第6回で追加した項目だけ消えるなら、保存項目や一覧カードへの表示が抜けている可能性があります。このように、原因を決めつけず、見る場所を分けます。

まず診断、すぐ修正しない

AIへの最初の一文


　まだ変更しないでください。
　まず確認すべき点を順番に教えてください。

原因を決めつけないことが大切。

保存できない、読み込めない、ログインできない、という状態になったとき、すぐに「直してください」と頼むと、AIが大きく変更してしまうことがあります。特にSecurity Rulesを広げすぎると危険です。そこで最初は、「まだ変更しないでください。まず確認すべき点を順番に教えてください」と聞きます。これは、AIを修理担当ではなく、点検の案内役として使う方法です。

診断プロンプト


　まだコードやSecurity Rulesを変更しないでください。
　まず、原因を決めつけず、確認すべき点を順番に整理してください。
　
　1. 直前にした操作:
　2. 画面に出た表示:
　3. エラー文:
　4. 期待した動き:
　5. 守りたい条件:
　　- Cloud Billing、Blaze、Cloud Run、Firebase App Hosting、Deploy、Publish、
　　　Get API key は使わない
　　- allow read, write: if true; にはしない
　　- ログインした本人の記録だけを扱う

このプロンプトは、すぐ修正させるのではなく、まず確認順を整理してもらうためのものです。授業中に不具合が出た場合、このプロンプトの空欄を埋めてAIに渡します。特に大事なのは、まだ変更しないでください、という一文です。これにより、AIがいきなりSecurity Rulesや保存処理を大きく変えてしまうことを防ぎます。

見せてはいけない情報

AIへの相談、スクリーンショット、画面共有では隠します。

  • メールアドレス
  • UID、ユーザー識別子
  • プロジェクトID
  • Database ID
  • APIキーのように見える文字列
  • 実名、勤務先、顧客情報
  • 業務上の機密情報
  • 支払い情報、請求情報

AIに相談するとき、スクリーンショットを撮るとき、画面共有をするときには、見せてはいけない情報があります。

メールアドレス、UID、ユーザー識別子、プロジェクトID、Database ID、APIキーのように見える文字列は、そのまま見せません。
