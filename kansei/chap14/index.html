なぜ今「ワークフロー」なのか

分析できたことと、伝わって使えることは別である。

たとえば、次の説明だけでは不十分。

  • t検定をした
  • 箱ひげ図を描いた
  • Bの平均がAより高かった

必要なのは、分析手法を 判断の根拠 に変えるワークフロー

  • 何を良くしたいのか
  • 誰の、どんな感じ方を測ったのか
  • どの図表・統計量から何が言えるのか
  • 結論の範囲と限界はどこか

感性データ処理では、分析結果は最終的に設計、改善、提案に使われます。したがって、分析ソフトを動かしただけでは十分ではありません。

読み手は、なぜその分析をしたのか、図から何が読み取れるのか、その結果から何を言ってよく、何を言い過ぎてはいけないのかを知りたいはずです。ワークフローは、その説明の筋道を作るための考え方です。

感性データ分析の7工程

  1. 問いを定める
  2. データの形を決める
  3. 前処理する
  4. 可視化する
  5. 分析する
  6. 解釈する
  7. レポートとして残す

重要なのは、工程名を覚えることではなく、
各工程で何を判断するかである。

第14回では、ワークフローを7工程で整理します。これは単に順番を暗記するためではありません。

それぞれの工程で判断することがあります。問いでは何を測るのか、データの形ではどの列を持つのか、前処理では何が怪しいのか、可視化では何を確認するのか、分析ではどの手法が問いに合うのか、解釈ではどこまで言えるのかを考えます。

ミニ活動1：何が足りないか

次の文を読み、結果としてまだ足りない情報を2つ挙げる。

「案Bは案Aより良かったので、案Bを採用する。」

考える観点

  • 何を「良い」と測ったのか
  • 誰が評価したのか
  • どの刺激・条件を比べたのか
  • 図や統計量は何を示したのか
  • 限界はどこか

Think 2分 → Pair 2分 → Share 1分

この活動では、結果文として不足している情報を見つけます。「良かった」という言葉には、好ましさ、使いやすさ、安心感、活発さなど複数の意味が混ざります。

また、誰が、どの条件で、どの尺度で、どの程度の差を示したのかが分からないと、採用という判断には進めません。ここから、問い、方法、結果、考察を分けて書く必要性につなげます。

生データを上書きしない

CSVを受け取ったら、最初にすることは保存である。

  1. data_raw/ などに原本CSVを置く
  2. 原本は編集せず、コピーまたはNotebookで整形する
  3. 整形後の表は data_processed/ などに保存する
  4. どの処理をしたかをメモする

避けたいこと

  • 原本CSVを直接開いて並べ替え、そのまま保存する
  • 手作業で値を直し、記録を残さない
  • 図用に別ファイルを作り、元データとの対応が分からない

分析でよくある事故は、生データを直接編集してしまうことです。あとから間違いに気づいても、元に戻せなくなります。

原本CSVは必ず保存し、整形や前処理はコピーやNotebook上で行います。手作業で直した場合も、何を直したのか記録が必要です。第15回の提出物でも、原本と処理済みデータの区別を意識してください。

コードブック：列の意味を残す

2刺激 × 3 VAS項目では、少なくとも次を説明できるようにする。

変数名 意味 値・単位 備考
respondent 回答者ID相当 R001, R002, … 個人名ではない
stimulus 刺激条件 A / B 同じ人が両方評価
order 提示順序 1 / 2 順序効果の確認に使う
item VAS項目 Q1 / Q2 / Q3 項目名と対応づける
value VAS値 0〜100 高いほど強い方向を明記

列名だけで意味が分からない場合、 未来の自分も読み直せない。

コードブックは、変数一覧です。列名、意味、単位、値の範囲、注意点を残します。

特にVASでは、値が高いほど何を意味するのかが重要です。好ましさなら高いほど好ましい、不快さなら高いほど不快、というように、方向を明記します。項目名とQ1、Q2、Q3の対応も必ず残してください。

Tidy data：分析しやすい表の基本形

分析しやすいデータは、次の形になっている。

  • 各変数が列
  • 各観測が行
  • 各セルに1つの値

VAS実験の縦長表

respondent stimulus order item value
R001 A 1 Q1 62
R001 A 1 Q2 48
R001 B 2 Q1 70
R001 B 2 Q2 42

可視化や集計では、 縦長表が扱いやすい。

Tidy dataは、分析しやすい表の基本形です。1つのセルに複数の情報を入れず、回答者、刺激、順序、項目、値を別々の列に分けます。

第13回でも、縦長表と横長表を扱いました。可視化や刺激別・項目別の集計には、縦長表が便利です。一方、同じ人のAとBの差分を作るときには、横長表へ変換することもあります。

横長表と縦長表を使い分ける

横長表：差分計算に向く

respondent A_Q1 A_Q2 A_Q3 B_Q1 B_Q2 B_Q3
R001 62 48 71 70 42 84

縦長表：可視化・集計に向く

respondent stimulus item value
R001 A Q1 62
R001 B Q1 70

レポートでは、どちらを使ったかではなく、 何を比べるために整形したかを説明する。

横長表と縦長表は、どちらが正しいというものではありません。目的によって使い分けます。

差分を作るときは、同じ回答者のAとBが同じ行にある横長表が便利です。刺激別、項目別に平均や箱ひげ図を作るときは縦長表が便利です。重要なのは、整形の目的を説明できることです。

前処理の目的

  • 何が測れていて、何が怪しいかを確認

最低限の点検

  • 回答数は想定に近いか
  • 欠損はどこにあるか
  • VAS値は0〜100に入っているか
  • 6評定がそろっているか
  • 刺激提示や再生の不具合はないか
  • 項目の方向はそろっているか

前処理は、結果をきれいに見せるために都合の悪いデータを消す作業ではありません。分析前にデータの状態を点検する作業です。

今回のVAS実験では、回答数、欠損、値の範囲、6つの評定値がそろっているか、不具合メモ、項目方向を確認します。点検した内容は、MethodsやResultsの補足として書けるようにします。

欠損があるときに確認すること

  • 欠損の件数と割合
  • どの刺激・どの項目に多いか
  • 途中離脱や表示不具合と関係していないか
  • 除外するのか、残すのか、その理由

無言で削除しない

欠損は、まず件数と場所を確認します。特定の刺激や項目だけ欠損が多い場合、質問文や刺激提示に問題があった可能性があります。

重要なのは、欠損を見つけた瞬間に削除するのではなく、何が起きているかを説明できる状態にすることです。全体で何件欠けているのか、AだけなのかBだけなのか、特定のVAS項目だけなのかを確認します。

レポートでは、除外した場合も残した場合も、その判断理由を書きます。無言で削除すると、結果がどのデータに基づいているのか読み手が追えなくなります。

2刺激 × 3項目で特に見ること

  • 1人につき6評定がそろっているか
  • Aだけ欠けている人、Bだけ欠けている人はいないか
  • 対応あり比較に使える人数はいくつか

欠損は、消す前に 把握して報告する。

今回の標準設計では、同じ回答者がAとBの両方を評価するため、対応あり比較に使える人数が重要です。AとBのどちらかが欠けると、その項目の差分には使えません。

たとえば、好ましさではAとBがそろっていても、落ち着きではBだけ欠けている人がいるかもしれません。その場合、項目ごとに分析に使える人数が変わります。

したがって、単に回答者数を書くのではなく、各項目でAとBのペアが何組そろっているかを確認します。これは、平均差や対応あり検定を読む前の前提条件です。

外れ値：本物の個人差か、手順違反か

外れ値には2種類ある。

種類 例 対応
本物の個人差 ある人だけ強く不快に感じた すぐ削除しない
入力ミス・手順違反 全項目100連打、条件取り違え 原因を確認する

感性データでは個人差が大きく出ることがあります。外れ値に見える値が、実はその人の本当の感じ方であることもあります。

一方で、0や100を連打している、刺激を見ていない、条件を取り違えたといった手順違反の可能性もあります。削除する場合は、結果を見てから都合よく決めるのではなく、基準と理由を明記します。

外れ値処理の基本手順

  1. まず図で見つける
  2. 原因を考える
  3. 除外するなら基準を明記する

「都合の悪い値だけ消す」
は避ける。

外れ値処理では、まず図で値の分布を確認します。箱ひげ図や散布図、差分図を見ると、他の回答から大きく離れた値や、特定の回答者だけ極端な差を示している場合に気づきやすくなります。

次に、その値が本物の個人差なのか、入力ミスや手順違反なのかを考えます。感性評価では、人によって強い反応が出ること自体は珍しくありません。

除外する場合は、結果を見てから都合よく消すのではなく、あらかじめ説明できる基準を置きます。レポートには、何を外れ値とみなし、何件をどの理由で除外したのかを書きます。

項目の方向をそろえる

VAS値が高いほど何を意味するかを確認する。

例

項目 高い値の意味
好ましさ より好ましい
落ち着き より落ち着く
活発さ より活発に感じる
不快さ より不快

VASの値が高いことの意味は、項目ごとに異なります。好ましさは高いほど肯定的ですが、不快さは高いほど否定的な反応です。

そのため、平均値を読む前に、各項目の方向を確認します。値が高いほど好ましいのか、落ち着くのか、活発なのか、あるいは不快なのかを明記しないと、同じ「高い」という結果でも解釈が逆になることがあります。

レポートでは、Methodsの尺度説明やコードブックに、各項目のアンカーと高得点の意味を残してください。

注意

  • 3項目を勝手に合計点にしない
  • 「好ましさ」と「不快さ」は方向が違う
  • 逆向き項目を処理した場合は、別列として保存する

したがって、3項目を何となく平均して総合点にしてはいけません。反転処理をする場合は、元の列を上書きせず、反転後の列を別名で保存します。レポートには、どの方向で読んだかを明記します。

特に、「好ましさ」と「不快さ」のように意味の方向が違う項目をそのまま足すと、何を測っている点数なのか分からなくなります。合計点を作るなら、項目が同じ概念を測っているという根拠と、方向をそろえる処理が必要です。

今回の標準課題では、まず3項目を別々に読み、どの項目でAとBの違いが見えたのかを説明する方針にします。

手作業を減らす

再現性を下げやすい操作

  • Excelで並べ替えて、そのまま上書きする
  • コピペで値を貼り直す
  • 図を作るために別ファイルを手修正する
  • 不要そうな行を目視で消す

手作業そのものが禁止というわけではありません。ただし、手作業は後から追跡しにくいことが多いです。

たとえば、Excelで並べ替えたあとにそのまま上書きすると、元の順序や変更前の値が分からなくなることがあります。コピペで値を直す操作も、どのセルを変えたのかが記録に残りにくいです。

分析結果に関わる操作ほど、あとから説明できる形にしておく必要があります。手作業を減らす目的は、楽をすることだけでなく、ミスを見つけやすくすることです。

望ましい形

  • Notebookやスクリプトで処理する
  • 手作業をした場合は、作業メモに残す
  • 原本CSVと処理済みCSVを分ける
  • 図の作り方を再実行できるようにする

特に、並べ替え、コピペ、目視削除は事故の原因になります。可能な限りNotebookやスクリプトで処理し、どうしても手作業が必要な場合は何をしたかメモに残します。

原本CSVと処理済みCSVを分けておけば、処理に失敗しても最初からやり直せます。図も、手作業で画像を作るのではなく、Notebookを再実行すれば同じ図が出る状態にしておくと安全です。

第15回で最終レポートを整えるときにも、どのデータからどの図を作ったのかが分かると、修正がかなり楽になります。

可視化：統計の前に状況を把握する

役割

  • 分布の偏りを見る
  • 床効果・天井効果を見る
  • 外れ値を確認する
  • 条件差の方向を確認する
  • 同じ人の差分を見る

先に図を見ることで、検定結果を誤解しにくくなる。可視化は、見栄えのためだけではない。

第13回でも確認したように、VASデータでは平均値やp値だけでは不十分です。同じ平均差でも、全員が少しずつ変化している場合と、一部の人だけが大きく変化している場合では意味が違います。

可視化は、統計解析の前にデータの状態を確認する安全点検です。

今回の主役は、 分布図 + 差分図 + 結果表である。

図は問いによって選びます。分布を見たいならヒストグラムや箱ひげ図、条件差を見たいなら箱ひげ図や平均と信頼区間、同じ人の変化を見たいなら差分図が向いています。

今回の主分析は広げすぎない

第12〜13回の標準仕様

  • 2刺激
  • 3 VAS項目
  • 同じ回答者がAとBを評価
  • 属性は原則取得しない

したがって

  • 各項目を別々に読む
  • B−A の差分を作る
  • 記述統計・分布図・効果量を示す
  • 必要に応じて対応のあるt検定またはWilcoxonを使う
  • 属性別比較や複雑な多変量解析は主分析にしない

今回のデータでは、分析を広げすぎないことが重要です。属性を原則取得していないので、属性別比較はできません。3つのVAS項目も、同じ尺度の下位項目とは限らないため、勝手に合計点にしません。

項目ごとにAとBを比較し、主分析項目を中心に結果を説明します。

Results と Discussion を混ぜない

節 書くこと 例
Results 図表で示された観測事実 案Bは案AよりスムーズさVASが高かった
Discussion 結果の意味づけ・限界・提案 ボタン配置の違いが寄与した可能性がある

注意

  • 「差があった」からすぐ「採用すべき」と書かない
  • Resultsでは、まず数値・図表・方向を書く
  • Discussionで、理由・限界・次の改善を書く

レポートで最もよく起きる失敗は、結果と考察が混ざることです。Resultsでは、図表や統計量から何が示されたかを書きます。Discussionでは、その結果がなぜ起きた可能性があるか、どこまで一般化できるか、次に何を改善すべきかを書きます。

採用や改善提案は、観測事実の後に、限界を踏まえて慎重に述べます。

結果段落の型

  1. どの図表を見るかを示す
  2. 重要な数値を1つ書く
  3. 図から何が言えるかを1文で述べる

例

図2に示すように、Q2「落ち着き」では刺激Aの方が刺激Bより高い傾向が見られた。回答者ごとの差分 B−A の平均は −13.5 点であり、効果量は … であった。この結果は、この2刺激の比較では刺激Aの方が落ち着きを強く感じさせたことを示す。

結果段落は、型を使うと書きやすくなります。まず図表を参照し、次に重要な数値を示し、最後に観測事実として何が言えるかを書きます。

この段階では、なぜ落ち着きが高かったのかという理由づけは控えめにします。それはDiscussionで扱います。

避けたい書き方と修正方向

避けたい書き方 修正の方向
Bの方が良かった どのVAS項目で高かったかを書く
有意差があったのでBが優れている 差の大きさ、図、限界も示す
p値が有意でなかったので差はない 効果量と不確実さを見る
この刺激は落ち着くと証明された 「この2刺激の比較では」と範囲を限定する
なんとなくBが好まれた 平均差・分布・自由記述など根拠を示す

主張は、測定した項目の範囲に限定する。

感性評価のレポートでは、「良い」「優れている」「証明された」といった強い表現を使いすぎないようにします。

今回測っているのは、3つのVAS項目です。したがって、どの項目についてどの方向の差があったのかを示します。結論は、今回の2刺激、今回の回答者、今回の手順の範囲に限定して書きます。

IMRAD形式で骨格を作る

節 何を書くか 今回のVAS実験での例
Introduction 背景、目的、研究質問 なぜ2刺激を比較するか
Methods 参加者、刺激、手順、尺度、分析方法 2刺激、3項目、VAS、対応あり比較
Results 図表、記述統計、検定結果 分布図、差分図、結果表
Discussion 意味づけ、限界、改善提案 なぜ差が出たか、どこまで言えるか
Conclusion 要点のまとめ 最も重要な結果と次の課題

IMRADは、読者が 問いから結論まで追いやすくする型である。

IMRAD形式は、Introduction、Methods、Results、Discussionの頭文字をとった構成です。最後にConclusionを加えると、授業レポートとして整理しやすくなります。

この形式を使うと、背景と問い、方法、結果、意味づけを分けて書けます。第15回では、この骨格に沿って最終レポートを完成させます。

Methods：方法で書くこと

今回のVAS実験では、少なくとも次を書く。

  • 参加者：授業内の学生、回答数、除外があれば理由
  • 刺激：刺激A・Bの内容、提示条件
  • 手順：提示順、回答割当、自分のグループには回答しないこと
  • 尺度：3つのVAS項目、アンカー、0〜100の意味
  • データ処理：欠損、外れ値、整形方法
  • 分析方法：記述統計、図、差分、効果量、検定

Methodsは、他者が 同じ手順を追える程度に書く。

Methodsは、実験の説明です。ここが曖昧だと、結果の意味も曖昧になります。

刺激AとBが何だったのか、3つのVAS項目が何だったのか、どのように回答したのか、どのようにデータを処理したのか、どの分析を行ったのかを書きます。長く書きすぎる必要はありませんが、読み手が手順を追えることが重要です。

Results：結果で書くこと

Resultsでは、解釈を急がず、観測事実を書く。

入れるもの

  • 回答数とデータ確認結果
  • 3項目の記述統計表
  • A/Bの分布図
  • 主分析項目の差分図
  • 平均差、95%信頼区間、効果量
  • 必要に応じて検定結果

書き方

  • 「図1に示すように…」
  • 「表1に示すように…」
  • 「平均差 B−A は…」

Resultsでは、まずデータ確認と結果を示します。図表番号を付け、本文で必ず参照します。

ここで大事なのは、結果を読み手に見せることです。図表を貼るだけではなく、どの図表のどこを見るのかを本文で説明します。

たとえば、「図1にA/B別の分布を示す」「表1に3項目の記述統計を示す」と書いたうえで、平均差やばらつき、差分の方向を述べます。必要に応じて、95%信頼区間、効果量、p値もここに置きます。

ただし、ここでは理由づけを急ぎません。「なぜそうなったか」「どちらを採用すべきか」はDiscussionで扱います。

Discussion：考察で書くこと

Discussionでは、結果の意味を考える。

書く観点

  • 主分析項目の結果は、仮説と一致したか
  • 刺激A・Bの違いが、結果にどう関係した可能性があるか
  • 3項目の結果は一貫しているか、分かれているか
  • 回答者数や刺激統制の限界は何か
  • 次に改善・追加検証するなら何をするか

注意

  • 因果を断定しすぎない
  • 結論の範囲を限定する
  • 予想外の結果も考察材料にする

Discussionでは、結果の意味を考えます。第12回で立てた仮説と結果を比べ、なぜそのような結果になった可能性があるかを述べます。

ただし、今回の授業内データだけで因果を強く断定することはできません。限界を明記し、そのうえで次にどう改善するかを書くと、考察が安定します。

図表の最低限ルール

  • 図表番号を付ける
    • 図1、図2、表1、表2
  • タイトルを付ける
  • 軸ラベル・単位・条件名を明記する
  • n数やエラーバーの意味を書く
  • 本文で必ず参照する
  • 何を伝える図なのかを先に決める

悪い状態

  • 図だけ貼って説明がない
  • 軸ラベルがない
  • A/Bが何を意味するか分からない

図表は、見た目を整えるだけではなく、読み手が誤解しないように作る必要があります。条件名、軸ラベル、単位、n数、エラーバーの意味が分からない図は、レポートの根拠として弱くなります。

図表は必ず本文で参照してください。「図2に示すように」と書くことで、読み手がどこを見ればよいか分かります。

再現性：未来の自分が再実行できる形にする

再現性は、立派な研究者だけの話ではない。

まず目指すこと

  • 数か月後の自分が、同じ結果を出せる
  • どのCSVからどの図を作ったか分かる
  • どの前処理をしたか分かる
  • どのNotebookを実行すればよいか分かる

再現性は、 未来の自分と他者への説明責任である。

再現性というと、大規模な研究公開を想像するかもしれません。しかし授業レベルでは、数か月後の自分が同じ結果を再現できることが出発点です。

どのCSVを使ったか、どの前処理をしたか、どのNotebookを実行したか、どの図がどの分析から出たかを追えるようにします。

  • どのファイルが生データか

  • VAS項目の方向や反転処理の有無

READMEは、 分析フォルダの取扱説明書である。

READMEは、分析フォルダの取扱説明書です。何の分析なのか、どのデータを使うのか、どのNotebookを実行するのか、どの図がどこに出るのかを書きます。

欠損や外れ値、反転処理のように、結果に影響する判断もREADMEに残しておくと、あとから確認しやすくなります。

大事なのはアルゴリズム単体ではなくパイプライン

データサイエンスでは、モデルや検定名に目が向きやすい。

しかし実務で価値になるのは、

  • どの問いから始めたか
  • どのデータをどう整えたか
  • どの図と分析で説明したか
  • その手順を他者が追えるか

という流れである。

データサイエンスでは、機械学習モデルや統計検定そのものに注目しがちです。しかし、実務で重要なのはパイプライン全体です。

どの問いから始めたのか、どのデータを集めたのか、どの前処理をしたのか、どの図表で説明したのかがつながっていなければ、結果だけが立派でも使いにくくなります。

今回のレポートでも、対応あり検定を使ったかどうかだけでなく、その検定に至るまでの測定、整形、可視化、解釈の流れを読者が追えることを目指します。

感性データでは特に

  • 質問文
  • VASのアンカー
  • 刺激提示条件
  • 回答文脈

が結果に影響する。

感性データでは、質問文、アンカー、刺激提示、回答文脈が結果に影響します。したがって、アルゴリズムだけでなく、測定と前処理と報告までを一つの流れとして考える必要があります。

同じ刺激でも、質問文が少し違えば回答の意味が変わることがあります。VASの左端と右端の表現、刺激を見せる順番、回答者がどのような状況で評価したかも、結果の読み方に関わります。

だからこそMethodsでは、刺激、手順、尺度を丁寧に書きます。Discussionでは、測定条件が結果に与えた可能性や限界も考察に含めます。

「きれいな図」より「誤解しない図」

図はプレゼン資料である前に、分析の道具である。

確認すること

  • 軸ラベルがあるか
  • 条件名が分かるか
  • VAS値の範囲が分かるか
  • 何人のデータか分かるか
  • 本文でその図を説明しているか

良い図は、 読み手が何を見ればよいか分かる図である。

図は見た目がきれいであることも大切ですが、それ以上に誤解しないことが重要です。

軸ラベル、条件名、VAS値の範囲、n数、本文での説明がそろっているかを確認します。とくにAとBの意味が分からない図は、レポートの読み手には伝わりません。

  1. 第14回は、新しい手法ではなく手法を使える形に束ねる回
  2. 分析は、問い → データ → 前処理 → 可視化 → 分析 → 解釈 → 報告で考える
  3. 生データを上書きせず、処理手順を残す
  4. VAS結果は、3項目を勝手に合計せず、項目ごとに読む
  5. Resultsでは観測事実、Discussionでは意味づけと限界を書く
  6. IMRAD形式で、最終レポートの骨格を作る
  7. READMEは、未来の自分が再実行するための取扱説明書である

今日のまとめです。第14回では、新しい統計手法を増やすのではなく、これまで学んだ手法を成果物へつなげる流れを整理しました。

特に、結果と考察を分けること、図表を本文で説明すること、データとNotebookを再実行できる形で残すことが重要です。

  • 図表の体裁を整える
  • Results を文章化する
  • Discussion と限界を書く
  • Conclusion をまとめる
  • 発表または提出用に最終確認する

最終目標は、 問い → 測定 → 分析 → 報告が一貫したレポートである。
